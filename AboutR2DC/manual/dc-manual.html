Introduction

MLP-DC is a visible, scalable, available tool for collecting, enriching, outputting log data. It also provides a beautiful web interface, make users be able to deploy, monitor everything on it.

Getting Started

Follow the steps to get a quick started:

  1. Install the mlp-dc distribution on somewhere, for example, /opt/mlp-dc.
  2. Start a node master on localhost: 
    /opt/mlp-dc/mlp-dc -m
  3. Start a node executor connecting to the master started on localhost: 
    /opt/mlp-dc/mlp-dc -e --master-ip localhost
  4. Check the node executor on the master RESTful API:
    curl -k https://localhost/v1/executors
  5. The executor started is now available on master side, learn following chapters and try to run some nodes on it!

Node Configuration

All node configuration should be formatted to JSON, the configuring operations are done on the web interface or by PUT into RESTful API, the basic node configuration format is:

{
    "NodeType": "FLUME",
    "Config": {}
}
  • NodeType is the type of the node.
  • Config is the config object.

Flume

Flume is one of the most popular open source data collection tools, MLP-DC integrated it natively. The main concept of Flume are 3 components: Source, Channel, Sink. Data is collected by source, pushed to channel, and flushed by sink.

Basic Options

Configuring a flume node is pretty simple in MLP-DC, let us start to understand a very simple configuration:

{
    "NodeType": "FLUME",
    "Config": {
        "Source": {
            "Type": "syslog",
            "Port": "514"
        },
        "Sink": {
            "Type": "logger"
        }
    }
}
  • NodeType must be assigned to FLUME here.
  • Config
    • Source is the input component of flume, For here, a syslog source that bound 514 port was set in this example.
      All supported sources:
      • syslog: Bind a port to receive UDP syslog packets. 
        "Source": {
              "Type": "syslog",
              "Port": "514"
          }
      • file: Read a single file. 
        "Source": {
              "Type": "file",
              "Path": "/opt/test.txt"
          }
      • dir: Read logs from a folder. This will be able to read the rolling logs. 
        "Source": {
              "Type": "dir",
              "Path": "/opt/"
          }
      • thrift: Bind a TCP ip & port to receive thrift logs. 
        "Source": {
              "Type": "thrift",
              "Port": "60000"
          }
      • last_node: Bind a Avro ip & port to receive avro logs, these logs are always from the last flume node. 
        "Source": {
              "Type": "last_node",
              "Port": "60000"
          }
    • Sink is the output component of flume. Here is one logger source.
      All supported sinks:
      • kafka: Send logs to kafka. 
        "Sink": {
              "Type": "kafka",
              "Broker": "127.0.0.1:12345",
        "Topic": "test_topic",
        "Compress": "true"
        }
      • next_node: Send logs to a opening avro source.
        "Sink": {
              "Type": "next_node",
              "HostName": "192.168.20.1",
        "Port": "60000"
        }
      • logger: Print received logs to the Flume logs.
        "Sink": {
              "Type": "logger"
        }
      • null: Drop all received logs.
        "Sink": {
              "Type": "null"
        }
Extra Options

The options follow are of default values, which means users can either configure them or just leave them default.

Option Default Value Description
ChannelSize 100000 The maximum number of logs the flume can cached, of unit event.
HeapSize 200 The JVM maximum heap size, of unit MB.
BatchSize 500 The grain in data transmission, of unit event/operation.
LogType "unknown-logtype" The log type.
LogTypeMapping N/A This option accept a JSON format to do mapping on log type.

Remote Linux Agent

The Remote Linux Agent Node is designed to collect log data from linux machines remotely and send logs to flume node.

Here is an example: to let the remote linux agent collect two machines file, then send to one flume remote source, write a configuration like this: 

{
    "NodeType":"REMOTE_LINUX_AGENT",
    "Config":{
        "flumeAddr":"192.168.20.55",
        "flumePort":8503,
        "flumeBatchSize":100,
        "sendCacheSize":1000,
        "localThriftServerPort":9996,
        "machines":{
            "192.168.20.58":{
                "user_name":"root",
                "machine_name":"58",
                "need_pause":"No",
                "password":"123dev",
                "log_type":{
                    "maillog":{
                        "file_reg":"/var/log/maillog*",
                        "is_multi_lines":false,
                        "charset":"utf-8",
                        "timestamp_reg":""
                    }
                }
            },
            "192.168.20.58":{
                "user_name":"root",
                "machine_name":"58",
                "need_pause":"No",
                "password":"123dev",
                "log_type":{
                    "maillog":{
                        "file_reg":"/var/log/maillog*",
                        "is_multi_lines":false,
                        "charset":"utf-8",
                        "timestamp_reg":""
                    }
                }
            }
        }
    }
}
  • flumeAddr, flumePort, flumeBatchSize: Configure the target flume server, flumeBatchSize means the numbers of event send to flume.
  • sendCacheSize: Events will be sent to flume every 1000 filling.
  • localThriftServerPort: Comming Soon
  • need_pause: Means if need to pause the thread when collecting the logs.
  • is_multi_lines
    • false means every event log only contains one line.
    • true means every event log contains multi lines.
  • timestamp_reg: The regular expression to split events.

RESTful API

Welcome

GET this URL to show the welcome text:

http://<Master-IP>/v1/

Below is the text received from the HTTP client:

Welcome to MLP DC API.

If you received this successfully, that means the master has been correctly started.

If you have some executors that has already connected to the master, the response will be like:

    "innerIP": "192.168.86.1",

    "errNum": 0,
    "nodeNum": 3,
    "osName": "Windows 7",

Executor Command

Users can send command to master to control one specified executor, to do this, GET a trigger using the path:

http://<master-IP>/v1/executors/<executor-UID>/command/<trigger>

From now on, the executor has just one trigger to send command:

  • refresh: Refresh a executor suddenly, in this operation, the master will perform a heartbeat request to executor, then will receive a new heartbeat from that executor with ignoring the heartbeat interval.
  • clear: Clear all reserved information about one executor locally on master side.

Node

In the case of multi nodes on one executor, users might want to show the details of one specified node, to do that, GET this path:

http://<master-IP>/v1/executors/<executor-UID>/nodes/<node-name>

The result will be the same as one entry in the result of getting all nodes details.

Node command

Node has several command to control:

  • update & add
    To add a new node, or replace an exist node, PUT your config to the path below:
    http://<master-IP>/v1/executors/<executor-UID>/nodes/<node-name>
    A reasonable configuration would be like (how to config nodes?):
    {
    "Type": "FLUME",
    "Source": {
      "Type": "File",
      "Path": "/var/logs/xxx.log"
    },
    "Collector": "192.168.22.123:8888",
    "BatchSize": "500",
    "HeapSize": "200"
    }
  • start & stop & delete & clear
    These 4 are triggers to control the life cycle of the nodes via using GET method on the path:
    http://<master-IP>/v1/executors/<executor-UID>/nodes/<node-name>/command/<trigger>
    • start: Start the node.
    • stop: Stop the node.
    • delete: Delete the node on executor side.
    • clear Clear the node informations locally on master side.

Development

Contents in this chapter is about to tell developers how to write plug-ins for MLP-DC.
MLP-DC is basically a data collection tool, moreover, it is also able to be a data processing framework, running different kinds of nodes on it, developers can make it support new node types by writing plug-ins for it. Take a look at belows to get started to plug-in writing.

Verbose Interface

Note: All development details are based on users have the web interface running. Plug-ins should be written in Java, and extend a Java Interface Node.

public interface Node {
    void init(String nodeName, String nodeType, String config, File dataDir) throws Exception;
 
    void start();
 
    void stop();
 
    void cleanup() throws NodeUntouchableException;
 
    boolean isRunning();
 
    NodeLog getNodeLog();
 
    int getBoundPort();
}
  • init(...): This method is invoked on the creation of the Node.
    • nodeName is given by the node name control on the Add New Node page of web interface.
    • nodeType is given by the configuration control on the Add New Node page of web interface.
    • config is given by the configuration control on the Add New Node page of web interface.
    • dataDir is the data folder provided by executor and for the node to save any files it needs.
  • start(): Start to run the node, invoked by the button Start on the web interface.
  • stop(): Stop the running node, invoked by the button Stop on the web interface.
  • cleanup() Clear all used files of this node, invoked by the button Clear on the web interface.
  • isRunning() Returns true if the node is running.
  • getNodeLog() Returns a instance of the Interface NodeLog.
  • getBoundPort() If the node has bound any port on this machine, return that port number.

Shortcut Interface

It is always unnecessary to implement the Node interface directly, as a shortcut, extend the AbstractNode class is a better way to write plug-ins.

public class TestNode extends AbstractNode {
    @Override
    protected void initNode() throws Exception {
 
    }
 
    @Override
    protected void stopNode() throws InterruptedException, IOException {
 
    }
 
    @Override
    protected void startNode() throws IOException {
 
    }
 
    @Override
    public void cleanup() throws NodeUntouchableException {
 
    }
 
    @Override
    public NodeLog getNodeLog() {
        return null;
    }
 
    @Override
    public int getBoundPort() {
        return 0;
    }
}

The methods are of the same use to the Interface Node, in the implementation of this class, developers no longer needs to bind the paras in init(...) into class, use get methods instead.

Utility Interface

  • NodeLog
    To monitor status of the node, running nodes should be able to report NodeLogs from its getNodeLog() method at any time:

    public interface NodeLog extends Serializable {
     
      List<ErrorEvent> getErrors();
     
      List<Metrics> getMetricsList();
     
      DestNode getDestNode();
     
      State getNodeState();
     
      String getType();
     
      enum State {
          READY, STOPPED, RUNNING
      }
    }
    • getErrors(): Returns all appeared errors between 2 invoking of getNodeLog().
    • getMetricsList(): Return all metrics between 2 invoking of getNodeLog().
    • getDestNode(): Return a flag of destination node of this log if sending logs to another node.
    • getNodeState(): Get a NodeState.
      • READY: First creation, have not run once.
      • RUNNING: Running.
      • STOPPED: Stopped.
    • getType(): Get the node type. If the node class extends the AbstactNode class, use the super method getNodeType() to fetch it.

  • Metrics Metrics is some numbers of resource usages of the node, Interface below:

    public interface Metrics extends Serializable {
      long getTimestamp();
     
      double getCpuUsage();
     
      long getMemUsed();
     
      long getEventTotal();
     
      long getEventSpeed();
    }
    • getTimestamp(): Returns a millisecond time stamp when this metrics be recorded.
    • getCpuUsage(): Returns a float number of the CPU usage of this node, 1.0 if 100%.
    • getMemUsed(): Returns a Byte memory use number of this node. For example, 50000 for 50 KB mem use.
    • getEventTotal(): Returns a event total number passed through this node since started.
    • getEventSpeed(): Returns current event speed, of unit event per second

Code And Integrate

Follow the steps to write and integrate a plug-in for MLP-DC:

  1. When starting to write a new node type plug-in, the first thing needed to do is to name the new type. Annotate the node class with NodeType to make the framework find the node class, put the node class into package path starting with prefix com.cloud.dc.node. The final node class should be something like:

    @NodeType("TEST")
    public class TestNode extends AbstractNode {
     @Override
     protected void initNode() throws Exception {
     
     }
     
     @Override
     protected void stopNode() throws InterruptedException, IOException {
     
     }
     
     @Override
     protected void startNode() throws IOException {
     
     }
     
     @Override
     public void cleanup() throws NodeUntouchableException {
     
     }
     
     @Override
     public NodeLog getNodeLog() {
         return null;
     }
     
     @Override
     public int getBoundPort() {
         return 0;
     }
    }
  2. Make unit test, make sure everything goes well.
  3. Package all Java classes into .jar file, put with all dependencies into mlp-dc/lib/
  4. Use following configuration to test the node:
    {
    "NodeType": "<My Node Type>",
    "Config": {}
    }
    All contents in field Config will be sent to node's init(...) method.
